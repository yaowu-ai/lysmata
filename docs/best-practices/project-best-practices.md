# WorkAny 项目最佳实践（可复用版）

> 目标：沉淀当前项目在 `Tauri + React + Hono + SQLite` 架构下可长期维护的工程实践，供其他项目直接借鉴。

## 1. 架构分层最佳实践

- **前端层（`src/`）**：只处理 UI、交互、状态和业务编排。
- **服务层（`src-api/`）**：承接业务规则和接口协议，不把复杂规则塞到组件中。
- **运行时层（`src-tauri/`）**：负责桌面能力、权限、安全边界、sidecar 生命周期与 SQLite 迁移。

核心原则：

- React 不承载原生系统职责。
- Rust 不承载前端业务流程。
- 前端与 API 通过 HTTP 解耦，便于替换和扩展。

---

## 2. Tauri 与 React 协作最佳实践

- **开发态**：`Tauri(WebView)` 加载 `http://localhost:1420`，API 独立运行在 `2026`。
- **生产态**：Tauri 加载打包静态资源，并在 Rust 中启动 API sidecar 到 `2620`。
- **端口策略固定**：`1420/2026/2620`，减少环境分歧和沟通成本。

建议：

- 把“环境差异”收敛在配置文件，不散落在业务代码中。
- sidecar 进程必须有启动日志与退出清理逻辑。

---

## 3. 数据层最佳实践（SQLite + fallback）

- **迁移归 Rust 管理**：数据库 schema 版本化，追加迁移而非覆盖历史。
- **读写归前端数据网关统一封装**：组件不直接写 SQL。
- **Web fallback**：非 Tauri 环境自动退回 IndexedDB，保证浏览器调试可用。

收益：

- 同一套业务代码可以在桌面与纯 Web 调试环境运行。
- schema 演进路径可追踪，可回放，可验收。

---

## 4. 配置与环境管理最佳实践

- 配置优先级明确：默认值 -> 配置文件 -> 环境变量 -> 运行时更新。
- API、provider、workspace 等配置统一由 config loader 管理。
- 关键常量集中管理（端口、目录、默认 provider、超时）。

建议：

- 禁止在代码中硬编码多处重复常量。
- 配置项命名统一，避免同义字段并存。

---

## 5. 安全与权限最佳实践

- `capabilities` 使用最小权限原则，只开业务必要能力。
- 文件系统权限按目录白名单收敛，不给全盘权限。
- shell 能力仅允许指定 sidecar 命令执行。

上线前必须确认：

- 权限范围与实际需求一致；
- 无“为方便调试临时放开权限”遗留。

---

## 6. API 设计与可维护性最佳实践

- 路由层做参数与协议，核心逻辑下沉到 `core`。
- 全局中间件统一日志、CORS、错误处理。
- 提供稳定健康检查端点（如 `/health`）用于联调与发布验证。

建议：

- 外部调用协议保持稳定，变更通过新增字段兼容。
- 为关键 API 提供最小 `curl` 验收脚本。

---

## 7. 工程与发布最佳实践

- Monorepo 用 `pnpm workspace` 管理根包和 `src-api` 子包。
- 开发命令分离：`dev:api`、`dev:app`，职责清晰。
- 打包链路明确：先构建 API 二进制，再构建 Tauri 桌面包。

建议：

- 发布脚本中显式区分平台产物，避免手工步骤。
- 每次发布至少执行一轮“安装包可启动 + API 可用 + DB 可迁移”验收。

---

## 8. 代码质量最佳实践

- 模块职责单一：UI、数据、服务、配置各归其位。
- 避免“胖组件”和“胖路由”，复杂逻辑拆分到独立模块。
- 统一格式化、Lint、类型检查规则，减少风格噪音。

建议：

- 关键流程补充日志点，便于定位跨层问题。
- 新功能默认附带回归检查清单。

---

## 9. 反模式清单（强烈避免）

- 在 React 组件直接拼接 SQL 或调用低层 Tauri API。
- 在 Rust 层堆业务规则，导致跨端演进困难。
- 生产依赖手工启动外部 API，而非 sidecar 托管。
- 权限一次性开太大，后续难收敛。
- 配置散落多个文件且无优先级定义。

---

## 10. 迁移到新项目的最小执行清单

1. 固化目录边界（`src` / `src-api` / `src-tauri`）。
2. 固化端口策略（`1420/2026/2620`）。
3. 接入 SQLite 插件并建立首版迁移。
4. 打通 `dev:api` + `dev:app` 双进程链路。
5. 配置 capabilities 最小权限。
6. 确认生产 sidecar 自动拉起与自动回收。
7. 跑通验收：`health`、任务写入、查询、打包启动。
