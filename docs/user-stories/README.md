## 产品北极星

Lysmata v1 的目标不是把 OpenClaw 的技术复杂度直接暴露给用户，而是帮助用户：

1. 安装并接入 OpenClaw
2. 用最少的显式配置完成首次成功
3. 通过模板快速创建第一个可用助手
4. 立刻开始第一次真实使用

一句话总结：

**Lysmata v1 是 OpenClaw 的易用层、首次成功工作台和多模板助手工坊。**

## 目标用户

### 主要用户

- **OpenClaw 意向用户**：想使用 OpenClaw，但不想被命令行、配置文件和底层概念卡住
- **轻技术用户**：能接受填写 API Key，但不希望自己搭系统和理解过多技术名词

### 次要用户

- **AI 开发者 / 高级用户**：需要更深的 Bot、Agent、Gateway 控制能力
- **研究者**：关注连接稳定性、持续运行和运行状态

> `完全非技术的大众用户` 不是 v1 的核心定义对象；v1 更适合服务“愿意使用 OpenClaw，但不想先学习 OpenClaw 内部结构”的用户。

## 用户故事格式

每条用户故事遵循以下格式：

```md
### US-{module}-{nn}：{标题}

**作为** {用户角色}，
**我希望** {功能描述}，
**以便** {业务价值}。

**优先级**：P0 / P1 / P2
**验收标准（AC）**：
- AC1：...
- AC2：...
```

---

## 优先级定义

- **P0**：没有它，用户无法完成首次成功或持续维护主路径
- **P1**：显著提升主要使用场景，但不阻断 v1 主路径
- **P2**：体验增强或高级能力，不影响 v1 上线判断

---

## v1 主线

v1 的默认用户路径应尽量接近：

1. 安装 OpenClaw
2. 显式填写 `LLM Key`
3. 进入模板工坊
4. 选择模板
5. 创建第一个可用助手
6. 直接开始第一次对话

这条主线要求：

- 前台少讲 `Gateway / Agent / Bot / Provider / Hooks`
- 后台自动完成尽可能多的默认配置
- 第一屏卖“结果”，后续再承接“系统”

---

## v1 模板策略

v1 首批模板固定为 3 个：

1. `通用助手`
2. `信息助手`
3. `日常任务助手`

模板策略采用“混合方式”：

- `通用助手` 作为最安全的默认入口
- `信息助手` 体现信息收集、整理、解释价值
- `日常任务助手` 体现 OpenClaw 不只是聊天，而是可以帮助完成目标

## 交付阶段

### Phase 0：首次成功地基

- 安装 OpenClaw
- 完成环境检查与错误解释
- 写入 `LLM Key`
- 默认 Gateway / Agent / Bot 可被后台自动组织

### Phase 1：v1 发布主线

- 多模板助手工坊首页
- 模板创建第一个助手
- 创建后直接进入私聊
- 私聊最小完成度补齐

### Phase 2：进阶控制台与扩展能力

- 群聊
- Artifact
- 技能市场
- 系统托盘
- 完整 Agent 控制台体验（v1 仅保留后置的最小后台能力）

## 术语

- `Gateway`：OpenClaw 的连接入口，负责承接运行时通信
- `Agent`：OpenClaw CLI 管理的本地运行实体，关注 workspace、绑定与运行时
- `Bot`：Lysmata 应用层的连接对象与助手实例载体
- `助手模板`：帮助用户快速创建第一个可用助手的前台产品入口

> 在 v1 中，用户首先看到的是 `模板` 和 `助手`，而不是 `Agent` 与 `Gateway`。
