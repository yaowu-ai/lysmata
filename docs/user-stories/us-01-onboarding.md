# US-01：安装与首次成功向导

**模块**：Onboarding
**对应设计**：`design/ui-onboarding.html`、`docs/prd-v2.1.md`
**对应实现**：`src/pages/Onboarding/`、`src/shared/store/wizard-store.ts`

## 模块定位

`US-01` 不再定义为“完整系统配置向导”，而是 v1 的 **首次成功向导**。

它的唯一目标是：

**让用户以最少的显式配置，安装 OpenClaw、填写 LLM Key、进入模板工坊，并开始创建第一个可用助手。**

因此 v1 的默认原则是：

- 前台尽量少暴露 `Gateway / Agent / Bot / Hooks` 等底层概念
- 用户显式完成的关键动作以 `LLM Key` 为主
- 其它必要参数尽量默认化、自动化、隐形化
- 高级配置后置到 `Settings` 或高级模式

## v1 主路径

```text
startup-guard(env-check)
  → intro
  → install
  → install-success
  → llm-key
  → template-select
  → first-assistant-create
  → first-chat-ready
```

> 对非技术用户而言，这条主路径应优先于任何“完整控制台”路径。

---

## 一、冷启动与安装

### US-01-00：首次启动自动进入首次成功向导

**作为** 新用户，
**我希望** 第一次打开 Lysmata 时自动进入首次成功向导，后续启动默认回到主界面，
**以便** 第一次使用有引导、后续使用不被重复打扰。

**优先级**：P0
**验收标准（AC）**：

- AC1：首次启动自动进入 `/onboarding`
- AC1.1：外层由 `StartupGuard` 自动完成一次环境判断；已安装用户默认回主界面，未安装用户进入 `/onboarding/intro`
- AC2：完成首次成功路径后写入完成标记，后续不再自动弹出
- AC3：开发和测试时可手动重置该状态

---

### US-01-01：欢迎页清楚说明“安装 OpenClaw，然后创建你的第一个助手”

**作为** 非技术用户，
**我希望** 在欢迎页看到清晰、结果导向的引导，
**以便** 我知道这不是在配置一个复杂系统，而是在准备创建第一个可用助手。

**优先级**：P0
**验收标准（AC）**：

- AC1：欢迎页标题与副标题明确强调“安装 OpenClaw”和“创建第一个助手”
- AC2：主按钮为“开始安装 OpenClaw”
- AC3：这一步没有上一步，也不能跳过
- AC4：欢迎页不展示高级系统概念说明

---

### US-01-02：安装前环境检查与失败解释

**作为** 首次安装的用户，
**我希望** 在安装前知道当前环境是否可继续，以及失败时看到可执行的提示，
**以便** 我能顺利完成安装，而不是卡在技术问题上。

**优先级**：P0
**验收标准（AC）**：

- AC1：由 `StartupGuard` 在进入 onboarding 前自动检测安装前必要条件
- AC2：以用户可理解的方式展示结果，而不是只显示技术日志
- AC3：失败时给出“如何修复”的解释
- AC4：进入安装后保留“继续安装 / 重试”的明确入口，而不是再展示独立 env-check 页面

---

### US-01-03：安装过程可见、可等待、可失败恢复

**作为** 非技术用户，
**我希望** 安装过程中看到进度和状态变化，并在失败时知道该怎么继续，
**以便** 我不会因为黑盒等待而中途放弃。

**优先级**：P0
**验收标准（AC）**：

- AC1：安装过程有进度反馈
- AC2：状态文案尽量使用用户语言，而不是纯终端输出
- AC3：失败时提供重试或返回上一步入口
- AC4：安装成功后自动进入下一阶段

---

### US-01-04：安装成功后进入最小配置，而不是进入系统控制台

**作为** 非技术用户，
**我希望** 安装完成后直接进入下一步“让助手可用”的配置，
**以便** 我能快速开始使用，而不是先理解系统架构。

**优先级**：P0
**验收标准（AC）**：

- AC1：安装成功页明确告诉用户“下一步只需完成最少配置”
- AC2：主按钮进入 `LLM Key` 配置步骤
- AC3：页面不要求用户先理解 Gateway、Agent、Bot 的关系

---

## 二、最小配置与首次成功

### US-01-05：向导只显式要求 LLM Key

**作为** 想尽快开始使用的用户，
**我希望** 在首次配置时只需要显式填写 LLM Key，
**以便** 我不用先做一堆技术判断就能继续。

**优先级**：P0
**验收标准（AC）**：

- AC1：首次配置主路径只强制要求 `LLM Key`
- AC2：用户可选择内置 Provider，并输入对应 API Key
- AC3：如果已有配置，页面自动预填并允许更新
- AC4：Gateway、默认 Agent、默认 Bot 等底层项不作为首次强制步骤
- AC5：若用户需要高级配置，页面以弱化方式提供“高级设置”入口

---

### US-01-06：模板选择成为首次成功的一部分

**作为** 初次使用的用户，
**我希望** 在完成 LLM Key 之后，直接进入模板选择，
**以便** 我能快速理解“接下来要创建什么助手”。

**优先级**：P0
**验收标准（AC）**：

- AC1：LLM Key 配置成功后直接进入模板选择页
- AC2：模板页默认展示 3 个模板：`通用助手`、`信息助手`、`日常任务助手`
- AC3：模板卡片用用户语言描述，不要求先理解底层系统结构
- AC4：模板选择属于首次成功主路径，不是高级功能

---

### US-01-07：根据模板创建第一个可用助手

**作为** 首次使用的用户，
**我希望** 选定模板后系统自动帮我创建第一个可用助手，
**以便** 我马上开始使用，而不是再手工配置 Bot。

**优先级**：P0
**验收标准（AC）**：

- AC1：模板创建流程尽量少输入，最多只要求极少量可选信息（如助手名称）
- AC2：系统自动完成默认 Gateway、默认 Agent、默认 Bot/助手实例等后台组织
- AC3：创建成功后用户无需回到空白管理页
- AC4：失败时给出用户可理解的错误提示和重试入口

---

### US-01-08：创建完成后直接进入第一次对话

**作为** 首次使用的用户，
**我希望** 第一个助手创建成功后自动进入第一次对话或可立即开始使用的页面，
**以便** 我真正完成“第一次用起来 OpenClaw”。

**优先级**：P0
**验收标准（AC）**：

- AC1：创建成功后默认进入对应助手的对话页，或给出唯一明确的“立即开始”按钮
- AC2：成功文案强调“你现在可以开始使用”，而不是“你现在可以去配置 Bot”
- AC3：首次成功以“开始对话”为终点，而不是“完成配置”为终点

---

## 三、可恢复与可维护

### US-01-09：中途退出后可继续完成首次成功

**作为** 用户，
**我希望** 在首次成功向导中途退出后，仍然能从主界面继续完成关键步骤，
**以便** 我不会因为中断一次就彻底失去引导。

**优先级**：P1
**验收标准（AC）**：

- AC1：退出向导后不应破坏已完成的步骤
- AC2：主界面应能提示用户继续完成首次成功路径
- AC3：再次进入向导时，已保存的 LLM 配置应自动预填

---

### US-01-10：重新运行向导时优先复用现有配置

**作为** 已经使用过 Lysmata 的用户，
**我希望** 重新运行向导时自动复用已有配置，
**以便** 我可以修正关键项，而不是从零开始重复输入。

**优先级**：P1
**验收标准（AC）**：

- AC1：重新进入向导时自动读取现有 LLM 配置
- AC2：必要默认项自动沿用当前系统状态
- AC3：重新运行向导的默认目标仍是“恢复首次成功路径”，不是“进入完整控制台”

---

## 四、高级入口与后置能力

### US-01-11：高级设置后置暴露

**作为** 进阶用户，
**我希望** 在首次成功主路径之外仍能访问高级配置能力，
**以便** 我在需要时调整 Gateway、Agent 或其它系统参数。

**优先级**：P1
**验收标准（AC）**：

- AC1：高级配置入口在首次路径中弱化展示
- AC2：高级入口不阻断默认主路径
- AC3：进入高级入口后，用户能继续前往 `Settings` 或相应高级模块

---

### US-01-12：Channel / Skills / Hooks 不进入首次成功主路径

**作为** 产品设计者，
**我希望** 将 Channel、Skills、Hooks 从首次成功主路径中后置，
**以便** 非技术用户不会被扩展能力打断。

**优先级**：P1
**验收标准（AC）**：

- AC1：首次成功主路径不强制出现 Channel、Skills、Hooks 多步配置
- AC2：若需要展示相关能力，应以“后续可扩展”或“高级能力”方式出现
- AC3：这些能力的真实配置闭环以后续模块和高级设置为准
