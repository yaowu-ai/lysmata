## 前端应用兼容

1. 本工程是一个桌面端应用，需要同时兼容Windows与MacOS
2. 应用中涉及到终端的操作，在MacOS中请以系统默认的SHELL($SHELL)为准
3. 应用中设计到终端的操作，在Windows中可以以这个优先级考虑 cmd git-bash PowerShell，如有必要，让用户自己选择
4. NodeJS中有很多兼容Windows与MacOS的方法及变量，请基于这些进行业务逻辑

## 工作流编排 (Workflow Orchestration)

### 1. 默认进入计划模式 (Plan Node Default)

- 遇到任何非微小任务（超过 3 个步骤或涉及架构决策时），必须先进入计划模式。
- 如果进展偏离预期，立即停下来重新制定计划——不要盲目推进。
- 计划模式不仅用于功能构建，也应当用于测试和验证阶段。
- 提前编写详细的技术规格说明，以尽可能消除歧义。

### 2. 子代理策略 (Subagent Strategy)

- 充分利用子代理（Subagent），以保持主会话的上下文窗口整洁。
- 将调研、探索和并行分析任务下放给子代理去完成。
- 面对复杂问题时，通过分配更多的子代理来增加算力投入。
- 每个子代理只负责一个单一方向，确保执行时的专注度。

### 3. 持续改进机制 (Self-Improvement Loop)

- 在收到用户的**任何**纠正后：将该问题模式记录并更新到 `tasks/lessons.md` 文件中。
- 为自己设定强制规则，避免犯同样的错误。
- 严格基于这些经验教训进行迭代，直到错误率显著下降。
- 在每次新会话开始时，主动回顾当前项目相关的经验教训。

### 4. 交付前必验证 (Verification Before Done)

- 在没有确凿证据证明代码能正常运行之前，绝不标记任务已完成。
- 在相关场景下，对比主干代码（main）和你的修改代码之间的行为差异。
- 扪心自问：“一位资深工程师 (Staff Engineer) 会批准 (Approve) 这段代码吗？”
- 运行测试，检查日志，用事实证明代码的正确性。

### 5. 追求优雅，兼顾平衡 (Demand Elegance)

- 对于非微小变动：停下来想一想，“还有更优雅的实现方式吗？”
- 如果当前的修复方案感觉像是临时补丁（hacky）：运用你现在掌握的所有知识，推倒重来，实现一个真正优雅的方案。
- 对于简单直接的修复，跳过此步骤——不要过度设计 (over-engineer)。
- 在提交工作成果之前，先自我审视和挑战你的代码。

### 6. 自主修复 Bug (Autonomous Bug Fixing)

- 收到 Bug 报告时：直接去修复它。不要等待手把手的指导。
- 自主定位日志、错误信息和失败的测试——然后解决它们。
- 做到让用户“零上下文切换”（即：独立搞定问题，不给用户增加认知负担）。
- 遇到 CI 测试失败时，主动去修复，不需要别人教你怎么做。

## 任务管理 (Task Management)

1. **计划先行**: 将带有待办检查项的计划写入 `tasks/todo.md`。
2. **验证计划**: 在开始编写代码前，先进行确认和沟通。
3. **追踪进度**: 在执行过程中，逐步勾选已完成的待办事项。
4. **解释变更**: 在每个步骤提供高层级的摘要说明，解释你做了什么。
5. **记录结果**: 在 `tasks/todo.md` 中添加复盘/评审结果。
6. **总结教训**: 在代码被纠正后，及时更新 `tasks/lessons.md`。

## 核心原则 (Core Principles)

- **简单至上 (Simplicity First)**: 让每一次修改都尽可能简单。将对现有代码的影响降到最低。
- **拒绝偷懒 (No Laziness)**: 深挖问题的根本原因。杜绝临时性的凑合修复。严格对标高级开发者的行事标准。
- **影响最小化 (Minimal Impact)**: 修改只应触及绝对必要的部分。极力避免在修复时引入新的 Bug。
