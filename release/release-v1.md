# Lysmata Release v1

**状态**：Aligned Draft
**更新时间**：2026-03-23
**来源**：基于 `release` 脑暴共识与重整后的 `docs/user-stories/`

## 一句话定义

**Lysmata v1 是 OpenClaw 的易用层、首次成功工作台和多模板助手工坊。**

它不是独立聊天产品，也不是把 OpenClaw 技术复杂度直接暴露给用户的系统控制台。

---

## 目标用户

v1 的核心对象不是“完全不懂 AI 的大众用户”，而是：

**想使用 OpenClaw，但不想被命令行、配置文件和底层概念卡住的人。**

这类用户的典型诉求是：

- 少碰命令行
- 少改 JSON / 配置文件
- 用图形界面和向导完成首次成功
- 先得到结果，再逐步理解系统

---

## v1 要解决的问题

当前最关键的两个问题是：

1. 用户装不上、配不起来 OpenClaw
2. 用户装好了之后，不知道怎么开始真正使用 OpenClaw

所以 v1 的核心目标不是展示更多系统能力，而是提升 **首日成功率**。

---

## 第一价值交付

v1 不应把“安装成功”当成终点，也不应把“配置成功”当成终点。

真正的价值交付是：

**用户第一次真正用起来 OpenClaw。**

对应的首次成功时刻是：

**用户通过模板创建出第一个可用助手，并立刻开始第一次真实对话。**

---

## v1 主路径

默认用户路径应尽量接近：

1. 安装 OpenClaw
2. 显式填写 `LLM Key`
3. 进入模板工坊
4. 选择模板
5. 创建第一个可用助手
6. 直接开始第一次对话

这条主路径要求：

- 前台少讲 `Gateway / Agent / Bot / Provider / Hooks`
- 后台自动完成尽可能多的默认配置
- 第一屏先卖“结果”，后面再承接“系统”

---

## 核心产品叙事

Lysmata v1 的主叙事是：

**基于 OpenClaw 的多模板助手工坊。**

前台承诺是：

- 我可以快速选一个我能理解的助手模板
- 我可以马上得到一个能用的助手
- 我不需要先学会 OpenClaw 的底层结构

后台真正完成的是：

- 替用户完成必要的 OpenClaw 接入与配置
- 替用户组织 Agent / Bot / Gateway 等底层对象
- 在用户成功之后，再逐步暴露更高级的系统能力

---

## 首屏策略

首页不应先展示复杂系统状态，也不应先展示大量技术概念。

v1 的默认信息架构为：

- 第一主区：推荐模板，帮助用户创建第一个助手
- 第二主区：最近使用 / 继续使用
- 第三主区：系统状态与配置入口（弱化展示）

设计目标是：

- 让第一次打开产品的用户知道下一步该做什么
- 让用户先获得价值，再理解系统
- 让 Lysmata 既是助手创建入口，也是 OpenClaw 的长期工作台

---

## 首批模板

v1 首批模板固定为 3 个：

1. `通用助手`
2. `信息助手`
3. `日常任务助手`

分工如下：

- `通用助手`：最安全的默认入口，保证用户先得到一个能用的助手
- `信息助手`：体现资料收集、整理、解释与汇总价值
- `日常任务助手`：体现 OpenClaw 不只是聊天，也能按步骤帮助用户完成目标

模板策略采用“混合方式”，既提供默认入口，又让用户感受到明确选择。

---

## 安装与首次配置策略

对非技术用户而言，安装配置向导不应继续承担“完整系统控制台”的职责。

v1 的默认策略是：

**向导原则上只让用户显式完成 `LLM Key` 配置，其它必要参数尽量默认化、自动化、隐形化。**

系统在后台自动处理的默认项包括：

- Gateway 默认地址与端口
- 默认 bind 策略
- 默认 Agent
- 默认 Bot / 助手实例
- 推荐 provider / model 映射
- 必要的健康检查与失败提示

同时保留弱化的高级入口，用于承接进阶用户，而不干扰默认首次成功路径。

## 交付阶段

### Phase 0：首次成功地基

- 安装 OpenClaw
- 环境检查与失败解释
- 写入 `LLM Key`
- 默认 Gateway / Agent / Bot 后台可组织

### Phase 1：v1 发布主线

- 多模板助手工坊首页
- 模板创建第一个助手
- 创建后直接进入私聊
- 私聊最小完成度补齐

### Phase 2：进阶控制台与扩展能力

- 群聊
- Artifact
- 技能市场
- 系统托盘
- 完整 Agent 控制台体验（v1 仅保留后置的最小后台能力）

---

## v1 必须成立的体验标准

如果以下任一项不成立，就不应宣称 v1 已经真正成立：

- 用户能顺利完成安装与最小配置
- 用户只需理解很少概念就能创建第一个助手
- 创建后能立刻进入第一次对话
- 对话体验足够顺滑，至少支持流式输出、停止生成、Markdown 可读、历史可继续
- 高级能力不会打断默认主路径

---

## 不应作为 v1 主叙事的方向

以下方向不应成为 v1 的主叙事：

- 纯消费者聊天产品
- 通用 AI 聊天桌面 App
- 以 Agent / Gateway / Provider / Hooks 为前台概念的系统控制台
- 先强调多智能体、Artifact、托盘等扩展能力，再考虑首次成功

---

## 仍待细化但不影响主线的事项

以下事项还可继续细化，但已经不影响 v1 的主方向：

- 首页模板卡片的具体文案与视觉层级
- 模板创建流程中的字段数量与默认值细节
- 最近使用区与系统状态区的版式关系
- 高级入口的命名与露出强度
